DIAGRAM 2: TELEMETRYMETRICS INITIALIZATION & USAGE FLOW
════════════════════════════════════════════════════════════════════════════════════

Purpose: Show how telemetry is initialized and how plugins use it

Decision Points (Diamonds):
  ◇ "Is Telemetry Enabled?" = Compile-time flag check
  ◇ "Initialize Method Called?" = Eager vs Lazy initialization
  ◇ "Publish?" = Record-only vs Record+Publish options

Two Initialization Paths:

Path 1: EAGER INITIALIZATION (RDKWindowManager, AppManager, LifecycleManager)
  ├─ initializeTelemetryClient() called during Initialize()
  ├─ Lock mutex: gTelemetryLock
  ├─ Query TelemetryMetrics interface
  ├─ Cache globally (static singleton)
  ├─ Log warning if not found (non-fatal)
  └─ Continue normally

Path 2: LAZY INITIALIZATION (Most other plugins)
  ├─ Skip initialization during Initialize()
  ├─ On first recordTelemetry() call:
  │  ├─ ensureTelemetryClient() checks cache
  │  ├─ If not cached: Query TelemetryMetrics
  │  ├─ Cache result
  │  └─ Use it
  ├─ Subsequent calls use cache (<1ms)
  └─ Works even if TelemetryMetrics starts later

Thread Safety:
  • std::mutex gTelemetryLock protects global cache
  • Double-check pattern prevents race conditions
  • Lock held only during query (<5ms)
  • Subsequent accesses don't need lock

Performance:
  • First call (query): ~3-5ms
  • Subsequent calls (cached): <1ms
  • No operational impact if unavailable

Read this diagram by:
  1. Start at "Plugin Initialize"
  2. Check if telemetry enabled
  3. Choose eager or lazy path
  4. Follow the locking sequence
  5. See query results (found/not found)
  6. Reach "TELEMETRY READY" state
  7. Record data via getTelemetryClient()

Color Groups:

  🌈 Multi-colored = Utility plugins

Central Hub: TelemetryMetrics
  Record(id, metrics, name)  = Store metrics data

Plugin Data Flow:

  • PackageManager (lazy-load)
    Records: Download time, Download errors
  
  • PreinstallManager

TIER 3:
  • LifecycleManager (lazy-load)
    Records: Launch time, Close time, State transitions

UTILITIES:
  • DownloadManager (lazy-load)

  • RDKWindowManager (eager-load)

Telemetry Data Flow:
  Plugin → ensureTelemetryClient() → Global Cache (locked)
         → Query if needed
         → TelemetryMetrics (record/publish)
         → Message Bus (external systems)
         → Analytics, Monitoring, Logging

DIAGRAM 4: SEQUENCE - WINDOWMANAGER & TELEMETRY - NO BLOCKING
════════════════════════════════════════════════════════════════════════════════════

Purpose: Show detailed execution sequence proving no thread blocking

Participants (Columns):
  Thunder Framework     = Main framework orchestrator
  RDKWindowManager      = The plugin being initialized
  TelemetryMetrics      = Telemetry service provider
  Shell Thread          = Background rendering thread
  TelemetryClient       = Telemetry client helper
  Global Cache          = Static gTelemetryMetricsObject

Sequence (Top to Bottom):

1. Thunder calls WindowManager.Initialize()
   └─ Plugin becomes active

2. WindowManager setup operations (non-blocking)
   ├─ Store IShell pointer
   ├─ Create event listener
   ├─ Enable inactivity reporting

3. Launch shell thread (non-blocking)
   └─ std::thread([=]() { ... }) returns immediately
   └─ Shell thread starts in background

4. Telemetry initialization attempt
   ├─ Lock mutex gTelemetryLock
   ├─ Check cache
   ├─ Query TelemetryMetrics (if needed)
   │  └─ May not be available yet (non-blocking query)
   ├─ Unlock mutex
   └─ Return success regardless

5. WindowManager.Initialize() returns (Complete!)
   └─ Main thread continues to next plugin

6. Shell thread runs independently
   └─ while(isRunning) {
        Lock mutex (briefly)
        Process display requests
        Render frame
        Unlock mutex
        Sleep for frame rate
      }

7. PARALLEL: Shell thread rendering loop continues

8. Later: TelemetryMetrics becomes available
   └─ Another plugin starts it

9. Later: First telemetry call from WindowManager
   ├─ ensureTelemetryClient() called
   ├─ Check cache: empty or populated?
   ├─ If empty: Lock, Query, Cache (fast)
   ├─ If populated: Use cached (instant)
   ├─ Record telemetry
   └─ Shell thread unaffected

10. Result: ✅ Everything works, no blocking!

Key Points Visualized:
  • Parallel execution (thick lines from parallel blocks)
  • Non-blocking operations (thin arrows)
  • Lock/Unlock sequences (activate/deactivate)
  • Alternative paths (alt blocks)
  • Timeline progression (top to bottom)

Thread Safety Highlighted:
  • Mutex protects global cache
  • Lock acquired only briefly during query
  • Shell thread continues rendering throughout
  • No blocking whatsoever

Read this diagram by:
  1. Start at top (Thunder framework)
  2. Follow WindowManager initialization (left column)
  3. Observe telemetry init (middle section)
  4. Note shell thread launching and running in parallel
  5. Skip ahead to see later telemetry call
  6. Verify no blocking occurs
  7. Conclusion at bottom
