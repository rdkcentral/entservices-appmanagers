DIAGRAM 2: TELEMETRYMETRICS INITIALIZATION - DETAILED WALKTHROUGH
════════════════════════════════════════════════════════════════════════════════════

What it shows:
  • Compilation flag check (ENABLE_AIMANAGERS_TELEMETRY_METRICS)
  • Two initialization paths (eager vs lazy)
  • Mutex lock/unlock sequence
  • Query success/failure handling
  • Data recording flow

Path A: DISABLED (Compile-time flag OFF)
───────────────────────────────────────

  [Plugin Initialize]
        ↓
  Is Telemetry Enabled?
        ↓ NO
  [DISABLED - All telemetry calls NO-OP]
        ↓
  [Continue Without Telemetry]


Path B: ENABLED - EAGER INITIALIZATION (RDKWindowManager, AppManager, LifecycleManager)
────────────────────────────────────────────────────────────────────────────────────────

  [Plugin Initialize]
        ↓
  Is Telemetry Enabled?
        ↓ YES
  [Store IShell pointer]
        ↓
  During Initialize():
  initializeTelemetryClient() called
        ↓
  Lock: std::mutex gTelemetryLock
        ↓
  Check global cache: gTelemetryMetricsObject?
        ├─ IF CACHED: Use it, unlock, continue
        └─ IF NOT CACHED:
            QueryInterfaceByCallsign("org.rdk.TelemetryMetrics")
            ├─ IF FOUND: Cache globally, unlock
            └─ IF NOT FOUND: Log warning, unlock, continue
        ↓
  Return Core::ERROR_NONE (regardless of telemetry result)
        ↓
  Plugin ready, telemetry available (or gracefully missing)


Path C: ENABLED - LAZY INITIALIZATION (Most other plugins)
────────────────────────────────────────────────────────────

  [Plugin Initialize]
        ↓
  Is Telemetry Enabled?
        ↓ YES
  [Store IShell pointer]
        ↓
  During Initialize():
  (Skip telemetry init - defer to first use)
        ↓
  Plugin ready
        ↓
  Later: recordTelemetry() called
        ↓
  ensureTelemetryClient() called
        ├─ IF AVAILABLE (cached): Use instantly (~0.5ms)
        └─ IF NOT AVAILABLE:
            initializeTelemetryClient() now
            Lock & Query (same as eager path)
            Cache result
            Unlock
            ~3-5ms on first call
        ↓
  Record telemetry
        ↓
  Success (or graceful failure if unavailable)


Performance comparison:
  Eager init failure:    Logged warning, app continues normally
  Lazy init:             No impact if TelemetryMetrics unavailable
  Cached access:         <1ms subsequent calls
  First cache miss:      ~3-5ms query, then cached

DIAGRAM 3: TELEMETRYMETRICS USAGE - DETAILED WALKTHROUGH
════════════════════════════════════════════════════════════════════════════════════

What it shows:
  • Each plugin's telemetry usage pattern
  • Data types recorded by each plugin
  • Flow from plugins to TelemetryMetrics
  • Message bus distribution to external systems

Hub: TelemetryMetrics (Central Service)
  ├─ Interface: ITelemetryMetrics
  ├─ Methods: Record(id, metrics, name), Publish(id, name)
  └─ Global cache: static gTelemetryMetricsObject

Spokes: Plugins using telemetry

EAGER LOADERS:
  • AppManager: Eager-load during Initialize()
    Records: App launch, close, crashes, errors
  
  • RDKWindowManager: Eager-load during Initialize()
    Records: Display creation time
  
  (Non-fatal if TelemetryMetrics unavailable)


LAZY LOADERS:
  • RuntimeManager: Load on first telemetry call
    Records: Container started, stopped, state changed
  
  • PackageManager: Load on first telemetry call
    Records: Download time, download errors
  
  • LifecycleManager: Load on first telemetry call
    Records: Launch time, close time, state transitions
  
  • DownloadManager: Load on first telemetry call
    Records: Download metrics, download errors
  
  (Works even if TelemetryMetrics starts after plugin)


NON-USERS:
  • PreinstallManager: No telemetry
  • OCIContainer: No telemetry
  • (Other external plugins): Varies per plugin


Data Flow:
  
  Plugin recordTelemetry()
        ↓
  Create JsonObject
        ↓
  Add metrics data
        ↓
  getTelemetryClient().record(appId, metrics, marker)
        ↓
  gTelemetryMetricsObject->Record(appId, metrics, marker)
        ↓
  TelemetryMetrics stores data
        ↓
  [Optional] Publish(appId, marker)
        ↓
  Message bus forwards to external systems
        ↓
  Analytics, Monitoring, Logging systems receive data

DIAGRAM 4: WINDOWMANAGER SEQUENCE - DETAILED WALKTHROUGH
════════════════════════════════════════════════════════════════════════════════════

What it shows:
  • Complete initialization sequence
  • Telemetry initialization attempt (non-blocking)
  • Shell thread launch and operation
  • Later telemetry connection
  • No blocking throughout

Timeline:

T+0ms: Thunder Framework calls WindowManager.Initialize(IShell*)
       ↓ Activate WindowManager column

T+1ms: Setup operations (millisecond range)
       ├─ mService = service
       ├─ mService->AddRef()
       ├─ Create event listener
       ├─ enableInactivityReporting()
       (All blocking together: <10ms)

T+5ms: Launch shell thread
       ├─ shellThread = std::thread([=]() { ... })
       ├─ Created in background
       └─ Main thread continues (non-blocking return)

T+10ms: Telemetry initialization attempt
        ├─ Lock gTelemetryLock (mutex)
        ├─ Check: gTelemetryMetricsObject cached?
        │  ├─ If YES: Use cached, unlock (~0ms)
        │  └─ If NO: QueryInterfaceByCallsign (~3-5ms)
        │     ├─ TelemetryMetrics may not be available yet
        │     ├─ Query returns null (non-fatal)
        │     └─ Unlock
        ├─ Log warning if not found
        └─ Continue

T+15ms: WindowManager.Initialize() returns
        └─ Thunder framework continues to next plugin

T+15-50ms: WindowManager rendering thread active
           └─ while(isRunning) {
                ├─ Lock gRdkWindowManagerMutex
                ├─ Process display creation requests
                ├─ Render frame (CompositorController draw)
                ├─ Unlock
                └─ Sleep for frame rate
              }
              (Runs independently forever)

T+100ms: TelemetryMetrics starts (in parallel initialization)
         └─ Interface becomes available

T+500ms: [Example] First telemetry call from WindowManager
         └─ recordCreateDisplayTelemetry() called
            ├─ ensureTelemetryClient()
            ├─ Check cache: gTelemetryMetricsObject?
            ├─ If cached: Use it (~0.5ms)
            ├─ If not cached: Query now
            │  ├─ Lock mutex
            │  ├─ QueryInterfaceByCallsign
            │  ├─ Find TelemetryMetrics ✓ (now available!)
            │  ├─ Cache it
            │  └─ Unlock (~3-5ms)
            ├─ Record telemetry
            └─ Success ✓

Result: ✅ NO BLOCKING OBSERVED
        ✅ Shell thread continues rendering throughout
        ✅ All operations non-blocking
        ✅ WindowManager fully independent

DIAGRAM 5: WINDOWMANAGER INTERACTIONS - DETAILED WALKTHROUGH
════════════════════════════════════════════════════════════════════════════════════

What it shows:
  • WindowManager central position
  • Connections to other plugins
  • Telemetry integration
  • Listener registration patterns

WindowManager (Central Blue Box):
  ├─ Initialize(): Shell thread launch, event setup
  ├─ Core Functionality: CreateDisplay, Screenshot, events
  └─ Shell Thread: Independent rendering loop

Connections:

1. WITH TELEMETRYMETRICS (Gold)
   ├─ Eager-load attempt during Initialize()
   ├─ Success or graceful failure
   └─ First telemetry call: record creation time

2. WITH RUNTIMEMANAGER (Green)
   ├─ RuntimeManager creates WindowManagerConnector
   ├─ Connector queries WindowManager interface
   ├─ Registers WindowManagerNotification listener
   └─ Receives inactivity & disconnection events

3. WITH LIFECYCLEMANAGER (Purple)
   ├─ LifecycleManager queries WindowManager
   ├─ Registers WindowManagerNotification listener
   ├─ Receives inactivity & disconnection events
   └─ Broadcasts to application layer

Data Flows:

RDK WindowManager → CreateDisplay request ← RuntimeManager
  └─ Runtime requests display creation
  └─ WindowManager shell thread queues request
  └─ Display created asynchronously

WindowManager → Inactivity event → LifecycleManager
  └─ User inactivity detected
  └─ LifecycleManager notified
  └─ Application lifecycle adjusted

WindowManager → Telemetry data → TelemetryMetrics
  └─ Display creation metrics recorded
  └─ Forwarded to external analytics
